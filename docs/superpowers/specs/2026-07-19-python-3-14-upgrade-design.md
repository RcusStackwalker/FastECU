# Python 3.14 Upgrade Design

## Objective

Upgrade FastECU from Python 3.11 to Python 3.14 as its sole supported Python
runtime. Keep Bazel as the source of truth for the hermetic interpreter and
align Ruff's language target with that runtime.

The work starts from `master` at commit
`0278bdc5192aa64b20d50e98d172acfc1f06ab39`.

## Scope

- Change every Bazel Python version declaration from 3.11 to 3.14.
- Import and use the generated `python_3_14` toolchain repository.
- Resolve the existing pip dependency set for Python 3.14.
- Change Ruff's target version from `py311` to `py314`.
- Regenerate and commit Bazel's module lockfile.
- Make only narrow script changes required by Python 3.14 or Ruff checks.

The upgrade does not add a multi-version test matrix, retain Python 3.11
compatibility, or update unrelated dependency and tool versions. In particular,
`rules_python` 2.2.0, PyYAML 6.0.3, and Ruff 0.15.14 remain pinned because their
current releases support the requested configuration.

## Configuration Architecture

`MODULE.bazel` remains the authoritative runtime configuration. Its
`python.defaults`, `python.toolchain`, imported toolchain repository, and
`pip.parse` declarations will all use Python 3.14. The existing minor-version
pinning model is retained, allowing `rules_python` to select its supported
3.14 patch release.

Bazel supplies this interpreter to all `py_binary` and `py_test` targets.
`pip.parse` resolves the existing hashed `requirements.txt` for the same Python
minor version, so Bazel-managed programs do not depend on a locally installed
Python.

Ruff remains a pre-commit tool rather than a Bazel dependency. Its
`target-version = "py314"` setting permits Python 3.14 syntax and makes its
version-aware upgrade rules consistent with the runtime.

## Dependency Compatibility

The direct third-party runtime dependency is PyYAML 6.0.3. It declares Python
3.14 support and publishes CPython 3.14 wheels for the project's Linux, macOS,
and Windows platforms. The existing `rules_python` 2.2.0 release also provides
Python 3.14 toolchains. No dependency upgrade is necessary to raise the runtime
ceiling.

The implementation will keep the checked-in, hash-pinned `requirements.txt`
unchanged unless Bazel demonstrates that its current hashes cannot resolve the
same PyYAML release under Python 3.14.

## Lockfile and Failure Handling

`MODULE.bazel.lock` will be regenerated by Bazel after the module configuration
changes; it will not be edited manually. Expected changes include the selected
Python toolchain metadata and Python-version-specific pip resolution state.

If Python 3.14 toolchain or wheel resolution fails for a supported platform, the
upgrade stops. It must not silently fall back to a system interpreter, a source
build, or a mixture of Python 3.11 and 3.14 declarations. Python or Ruff
findings exposed by the upgrade may be fixed narrowly, while unrelated
formatting and refactoring remain out of scope.

## Verification

The implementation will:

1. Confirm the Bazel-managed interpreter reports Python 3.14.
2. Run every Bazel `py_test` target and Python-backed validation target.
3. Verify PyYAML imports through a Bazel target under the Python 3.14 toolchain.
4. Run the full maintained Bazel test suite to detect integration regressions.
5. Run Ruff lint and formatting checks through the repository's pre-commit
   configuration.
6. Search project configuration for stale Python 3.11 and Ruff `py311` pins.
7. Review the final diff for regenerated state and unrelated changes.

Checks that require other operating systems remain the responsibility of the
existing Linux, macOS, and Windows CI jobs. Local verification must run every
check supported by the current host.

## Success Criteria

The upgrade is complete when Bazel and Ruff consistently target Python 3.14,
the hermetic interpreter and PyYAML-backed targets execute successfully, all
locally runnable checks pass, cross-platform CI passes, and the regenerated
lockfile contains no accidental mixed-version configuration.
